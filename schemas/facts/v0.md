# Kasane canonical fact schema v0

Schema v0 is the first **canonical extraction schema** for Kasane.
It is the boundary between frontend extraction and analysis.

The design goal is to keep the stored facts:

- **build-aware**, so the same source file compiled with different flags can be represented as different translation units,
- **relation-first**, so analyses are not forced to depend on a single in-memory graph layout,
- **small but expressive**, so local flow, lightweight interprocedural summaries, SARIF traces, and a derived CPG view can all be built on top of it,
- **incremental-friendly**, so TU-level caching and function-level summary caching are possible later.

This document defines the logical schema, invariants, and serialization rules.
It does **not** define derived analysis relations such as taint, local flow, points-to, or a persistent graph database layout.

## Scope

Schema v0 is intended to support:

- build-aware extraction from `compile_commands.json`,
- source navigation and finding location reporting,
- AST reconstruction for function bodies,
- CFG construction for intraprocedural analysis,
- direct call modeling,
- variable / field references, assignments, returns, and literals,
- type attachment at declaration and expression sites,
- preprocessor context needed for diagnostics and invalidation.

Schema v0 intentionally does **not** try to encode:

- full PDG / DFG as stored facts,
- whole-program points-to,
- template metaprogramming internals beyond what Clang materializes into declarations/types/nodes,
- path feasibility constraints,
- cross-run persistent IDs as primary keys.

## Identity model

Kasane uses two identity layers.

### 1. Snapshot-local opaque IDs

Extraction emits opaque string IDs for the current snapshot.
These IDs are only required to be unique **within one extraction result**.
They are cheap to generate, easy to debug in text fact files, and ideal for joins.

The main ID spaces in v0 are:

- `file_id`
- `tu_id`
- `macro_id`
- `type_id`
- `decl_id`
- `node_id`
- `expansion_id`

The value `"-"` is reserved as a null / unresolved sentinel for optional foreign keys.

### 2. Cross-run stable fingerprints

Cross-run continuity is handled separately via `fingerprint(entity_space, entity_id, stable_hash)`.

That split is deliberate:

- local IDs optimize joins and extraction speed,
- stable fingerprints optimize alert deduplication, summary cache reuse, and explainability.

Source ranges are **not** identities. They are reporting metadata.

## Schema versioning convention

Kasane chooses a **directory / file versioning convention** for canonical fact
schemas.

- The schema version is identified by the contract files under
  `schemas/facts/`, such as `schemas/facts/v0.md` and `schemas/facts/v0.dl`.
- A fact set targeting this schema should be treated as a `v0` snapshot and
  stored under a schema-specific directory such as `<facts_root>/v0/`.
- Within that directory, each relation is serialized as `<relation>.facts`.

This keeps relation payloads free of a repeated `schema_version` column while
still making breaking upgrades explicit at the dataset boundary.

## Serialization rules

The bootstrap text format is UTF-8 TSV.

- Each relation is stored in a file named `<relation>.facts`.
- Columns are tab-separated.
- Strings are escaped with C-style escapes for `\\`, `\t`, and `\n`.
- Booleans are stored as `0` or `1`.
- Opaque IDs are serialized as escaped strings.
- Byte ranges are half-open: `[start_byte, end_byte)`.
- Line and column numbers are 1-based.
- `end_column` is exclusive when available.

A future binary encoding may be added later, but the logical schema below is the source of truth.

## Relation overview

Schema v0 is split into five groups:

1. build and source context,
2. type and declaration inventory,
3. syntax and control-flow nodes,
4. semantic primitives for analysis,
5. source locations, stability, and diagnostics.

## Relation definitions

### 1. Build and source context

#### `file(file_id, path, content_sha256)`

One row per source file or included header observed by extraction.

- `file_id:symbol`
- `path:symbol` — normalized path as seen by the extractor
- `content_sha256:symbol` — content hash, recommended form `sha256:<hex>`

#### `translation_unit(tu_id, root_file_id, working_dir, compiler, target_triple, command_hash)`

One row per compilation command / TU.

- `tu_id:symbol`
- `root_file_id:symbol`
- `working_dir:symbol`
- `compiler:symbol`
- `target_triple:symbol`
- `command_hash:symbol` — normalized compile command hash

#### `compile_arg(tu_id, arg_index, arg)`

Lossless ordered view of the compile command used for the TU.

- `tu_id:symbol`
- `arg_index:number` — dense, zero-based
- `arg:symbol`

#### `tu_file(tu_id, file_id)`

Membership relation between a TU and any file it directly or indirectly consumed.

- `tu_id:symbol`
- `file_id:symbol`

#### `include_edge(src_file_id, dst_file_id, line, is_system)`

Preprocessor include relationship.

- `src_file_id:symbol`
- `dst_file_id:symbol`
- `line:number`
- `is_system:number` — `1` for `<...>` includes, else `0`

#### `macro(macro_id, name, is_function_like, arity)`

Macro definitions visible to the extractor.
Location is carried by `span(entity_space = "macro", ...)`.

- `macro_id:symbol`
- `name:symbol`
- `is_function_like:number`
- `arity:number` — `0` for object-like macros

#### `macro_expansion(expansion_id, macro_id)`

Macro expansion event.
Location is carried by `span(entity_space = "macro_expansion", ...)`.

- `expansion_id:symbol`
- `macro_id:symbol`

### 2. Types and declarations

#### `type(type_id, kind, printed, canonical, size_bits, align_bits)`

Canonicalized type inventory.

- `type_id:symbol`
- `kind:symbol` — e.g. `builtin`, `pointer`, `record`, `array`, `function_proto`, `reference`, `typedef`, `enum`, `unknown`
- `printed:symbol` — user-facing spelling from the frontend
- `canonical:symbol` — canonical type spelling
- `size_bits:number` — `0` if unavailable
- `align_bits:number` — `0` if unavailable

#### `type_edge(type_id, role, slot, other_type_id)`

Structural relationship between types.

- `type_id:symbol`
- `role:symbol` — recommended values: `pointee`, `element`, `return`, `param`, `underlying`, `base`
- `slot:number` — parameter index for `role = "param"`, else `0`
- `other_type_id:symbol`

#### `decl(decl_id, tu_id, kind, name, qualified_name, usr, type_id, parent_decl_id)`

Master declaration table.

- `decl_id:symbol`
- `tu_id:symbol`
- `kind:symbol` — recommended values: `function`, `param`, `global_var`, `local_var`, `field`, `record`, `typedef`, `namespace`, `enum`, `enum_const`
- `name:symbol`
- `qualified_name:symbol`
- `usr:symbol` — frontend-specific stable symbol if available, else empty string
- `type_id:symbol` — `"-"` if unavailable or not applicable
- `parent_decl_id:symbol` — lexical / semantic parent, or `"-"`

#### `function_decl(decl_id, mangled_name, is_definition, linkage, storage, is_variadic)`

Function-specific attributes.

- `decl_id:symbol`
- `mangled_name:symbol`
- `is_definition:number`
- `linkage:symbol` — e.g. `external`, `internal`, `unique_external`, `none`
- `storage:symbol` — e.g. `static`, `extern`, `auto`, `none`
- `is_variadic:number`

#### `param_decl(decl_id, function_decl_id, param_index)`

Function parameter placement.

- `decl_id:symbol`
- `function_decl_id:symbol`
- `param_index:number` — dense, zero-based

#### `global_var_decl(decl_id, tu_id, storage, linkage, has_initializer)`

Global or namespace-scope variable attributes.

- `decl_id:symbol`
- `tu_id:symbol`
- `storage:symbol`
- `linkage:symbol`
- `has_initializer:number`

#### `local_var_decl(decl_id, function_decl_id, storage, has_initializer)`

Function-local variable attributes.

- `decl_id:symbol`
- `function_decl_id:symbol`
- `storage:symbol`
- `has_initializer:number`

#### `record_decl(decl_id, tag_kind, is_definition)`

Struct/class/union metadata.

- `decl_id:symbol`
- `tag_kind:symbol` — `struct`, `class`, `union`
- `is_definition:number`

#### `field_decl(decl_id, record_decl_id, field_index)`

Field ordering inside a record.

- `decl_id:symbol`
- `record_decl_id:symbol`
- `field_index:number` — dense, zero-based

### 3. Syntax and control-flow nodes

#### `node(node_id, tu_id, function_decl_id, category, kind, type_id)`

Master syntax node table.

- `node_id:symbol`
- `tu_id:symbol`
- `function_decl_id:symbol` — enclosing function, or `"-"` for TU-scope nodes
- `category:symbol` — recommended values: `stmt`, `expr`, `synthetic`
- `kind:symbol` — frontend AST class or synthetic marker, e.g. `IfStmt`, `BinaryOperator`, `DeclRefExpr`, `CallExpr`, `FunctionEntry`, `FunctionExit`, `InitAssign`
- `type_id:symbol` — expression type if known, else `"-"`

Synthetic nodes are allowed in v0 to represent stable CFG entry/exit points or normalized initialization assignments.

#### `ast_edge(parent_node_id, child_index, child_node_id, label)`

Ordered AST containment.

- `parent_node_id:symbol`
- `child_index:number` — dense, zero-based
- `child_node_id:symbol`
- `label:symbol` — optional semantic label such as `callee`, `lhs`, `rhs`, `cond`, `then`, `else`; empty string if unavailable

#### `cfg_edge(src_node_id, dst_node_id, edge_kind)`

Intraprocedural control-flow edge.

- `src_node_id:symbol`
- `dst_node_id:symbol`
- `edge_kind:symbol` — recommended values: `normal`, `true`, `false`, `back`, `exceptional`

### 4. Semantic primitives for analysis

#### `ref(node_id, decl_id, role)`

Reference from a syntax node to a declaration.

- `node_id:symbol`
- `decl_id:symbol`
- `role:symbol` — recommended values: `read`, `write`, `address`, `callable`, `base`

`ref` is intentionally primitive. Richer def-use chains are derived later.

#### `use(node_id, decl_id)`

Read-like use attached to an analysis-relevant node such as an assignment,
call, or return.

- `node_id:symbol`
- `decl_id:symbol`

`use` is a convenience relation for local flow rules.
It may be emitted alongside more precise expression-level `ref` rows.

#### `def(node_id, decl_id)`

Definition attached to an analysis-relevant node such as a local initializer or
assignment.

- `node_id:symbol`
- `decl_id:symbol`

Like `use`, this is a convenience relation intended to keep local data-flow
rules simple without losing the more primitive `ref` relation.

#### `assign(node_id, lhs_node_id, rhs_node_id, op_kind)`

Assignment-like operation.

- `node_id:symbol`
- `lhs_node_id:symbol`
- `rhs_node_id:symbol`
- `op_kind:symbol` — e.g. `=`, `+=`, `init`

This relation may refer to a synthetic `node` for normalized local initialization.

#### `call(node_id, callee_decl_id, callee_name, dispatch_kind)`

Call-like expression.

- `node_id:symbol` — must reference a `node` whose `kind` is a call-like node
- `callee_decl_id:symbol` — `"-"` if unresolved or indirect
- `callee_name:symbol` — best-effort textual callee name
- `dispatch_kind:symbol` — recommended values: `direct`, `member`, `virtual`, `indirect`, `constructor`

#### `call_arg(call_node_id, arg_index, arg_node_id)`

Ordered call arguments.

- `call_node_id:symbol`
- `arg_index:number` — zero-based
- `arg_node_id:symbol`

#### `return_expr(return_node_id, expr_node_id)`

Return statement / returned expression pairing used by local flow rules.

- `return_node_id:symbol`
- `expr_node_id:symbol` — `"-"` for bare `return;`

#### `return_value(return_node_id, expr_node_id)`

Return statement / returned expression pairing.

- `return_node_id:symbol`
- `expr_node_id:symbol` — `"-"` for bare `return;`

`return_value` remains available as a compatibility alias for early v0
consumers. New rules should prefer `return_expr`.

#### `type_of(node_id, type_id)`

Convenience attachment from a node to its recovered type.

- `node_id:symbol`
- `type_id:symbol`

This duplicates `node.type_id` intentionally so analyses that focus on a small
subset of semantic facts can import types without depending on the full `node`
table shape.

#### `literal(node_id, literal_kind, raw_text)`

Literal text attached to a node.

- `node_id:symbol`
- `literal_kind:symbol` — e.g. `integer`, `string`, `char`, `float`, `bool`, `null`
- `raw_text:symbol`

#### `member_access(node_id, base_node_id, field_decl_id, access_kind)`

Field/member access primitive.

- `node_id:symbol`
- `base_node_id:symbol`
- `field_decl_id:symbol` — `"-"` if unresolved
- `access_kind:symbol` — `dot`, `arrow`, `qualified`

### 5. Locations, stability, and diagnostics

#### `span(entity_space, entity_id, file_id, start_byte, end_byte, start_line, start_column, end_line, end_column)`

Source location for declarations, nodes, macros, or macro expansions.

- `entity_space:symbol` — recommended values: `decl`, `node`, `macro`, `macro_expansion`
- `entity_id:symbol`
- `file_id:symbol`
- `start_byte:number`
- `end_byte:number`
- `start_line:number`
- `start_column:number`
- `end_line:number`
- `end_column:number`

#### `fingerprint(entity_space, entity_id, stable_hash)`

Stable identity material for cross-run matching.

- `entity_space:symbol`
- `entity_id:symbol`
- `stable_hash:symbol` — recommended form `sha256:<hex>`

The exact fingerprint recipe is implementation-defined in v0, but it must be deterministic for the same source snapshot and compile context.

#### `diag(tu_id, file_id, line, column, severity, message)`

Frontend diagnostics captured during extraction.

- `tu_id:symbol`
- `file_id:symbol`
- `line:number`
- `column:number`
- `severity:symbol` — e.g. `note`, `warning`, `error`, `fatal`
- `message:symbol`

Diagnostics do not invalidate the schema. Partial extraction is allowed as long as emitted facts obey referential integrity.

## Invariants

Schema v0 imposes these invariants.

### General

- Every foreign key either references an existing row in the target relation or is `"-"` when explicitly documented as nullable.
- `compile_arg.arg_index`, `param_decl.param_index`, `field_decl.field_index`, `call_arg.arg_index`, and `ast_edge.child_index` are dense and zero-based within their parent scope.
- Paths are normalized before hashing and emission.
- The same `file_id` must not appear with two different `content_sha256` values in one snapshot.

### Declarations and types

- Every `function_decl.decl_id` must reference `decl(kind = "function")`.
- Every `param_decl.decl_id` must reference `decl(kind = "param")`.
- Every `global_var_decl.decl_id` must reference `decl(kind = "global_var")`.
- Every `local_var_decl.decl_id` must reference `decl(kind = "local_var")`.
- Every `field_decl.decl_id` must reference `decl(kind = "field")`.
- Every `record_decl.decl_id` must reference `decl(kind = "record")`.
- `decl.type_id = "-"` is allowed only when the frontend truly cannot recover a type.

### Nodes and flow

- Every `node.function_decl_id != "-"` must reference a function declaration from the same `tu_id`.
- Every `cfg_edge` connects nodes belonging to the same enclosing function.
- `call.node_id`, `assign.node_id`, `return_expr.return_node_id`,
  `return_value.return_node_id`, and `member_access.node_id` must reference
  rows in `node`.
- `ast_edge` must form a directed acyclic containment relation per TU.
- Synthetic nodes must use `category = "synthetic"` and a stable `kind` name.

### Spans and fingerprints

- Every `span` must refer to an entity that exists in the declared `entity_space`.
- For any given `(entity_space, entity_id)`, there should be at most one primary `span` row in v0.
- `fingerprint` values must be deterministic for the same extraction input.
- `span` is reporting metadata only and must not be used as the sole stable identity across runs.

## Recommended fingerprint strategy

The schema leaves the exact recipe open, but the recommended v0 strategy is:

- for declarations: frontend USR if available, otherwise a normalized tuple of `(parent fingerprint, kind, qualified_name, canonical type, compile context)`;
- for nodes: `(enclosing function fingerprint, kind, AST child path, normalized token spelling)`;
- for files: content hash plus normalized path.

In all cases, the hash payload should be versioned so the recipe can evolve without silent collisions.

## What analyses should derive rather than store

The following belong outside the canonical extraction schema:

- local def-use and SSA-like relations,
- local taint,
- interprocedural summaries,
- dominance / post-dominance,
- control dependence,
- points-to / alias sets,
- CPG overlays,
- SARIF findings.

Those can be derived in `analyses/` from v0 facts and cached separately if needed.

## Mapping from the current bootstrap demo

The repository currently includes a tiny bootstrap schema for `sample.cpp`:

- `func.facts`
- `param.facts`
- `assign.facts`
- `callsite.facts`
- `return_var.facts`

That bootstrap schema is not the canonical schema.
It maps to v0 conceptually like this:

- `func` → `decl(kind="function")` + `function_decl` + `span`
- `param` → `decl(kind="param")` + `param_decl`
- `assign` → `def` + `use` + `assign`
- `callsite` → `call` + `call_arg` + `span`
- `return_var` → `return_expr` + `ref`

The bootstrap demo stays useful for onboarding, but new extractors should target v0.

## Evolution policy

v0 is the minimum schema that should remain stable long enough to build:

- a build-aware extractor,
- local flow rules,
- summary-based interprocedural analysis,
- SARIF emission,
- an optional derived CPG view.

Rules for changing v0:

- adding nullable relations or nullable columns is allowed in a minor update,
- renaming relations or changing column order requires `v1`,
- changing semantic meaning without renaming is not allowed,
- extraction and analysis code must version-gate schema upgrades explicitly.
